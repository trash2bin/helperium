# AI Assistant in a Box — план миграции к pre-final версии

## Текущее состояние

Технически на руках уже нетривиальный набор: автогенерация тулов из SQL-схемы, децентрализованная логика в докере (сервисы разнесены по контейнерам, а не монолит), сам агент с tool calling и минимальный RAG. Это не заготовка с нуля — это уже рабочее ядро, которого у большинства no-code конкурентов на рынке просто нет (они дают конструктор сценариев, а не агента с автогенерацией доступа к произвольной БД). Проблема сейчас не в ядре, а в том, что вокруг него нет «поверхности», которую можно физически показать и продать клиенту.

## Что считать pre-final версией

Pre-final — это не «фичи из оригинального роадмапа все реализованы», а минимальный набор, которого достаточно, чтобы один раз довести до конца платное внедрение одному реальному клиенту без ручного костыляния на каждом шагу. Специализация под Битрикс24/1С/VK сюда сознательно не входит — это следующий этап, после того как эта универсальная версия хотя бы раз продалась и отработала на реальных данных.

## Что есть и чего не хватает

| Есть | Не хватает |
|---|---|
| Автогенерация тулов из SQL | Слой ревью сгенерированных тулов перед тем, как давать им доступ к боевой БД клиента |
| Децентрализованные сервисы в докере | Один чистый docker-compose + `.env.example`, по которому разворачиваешь у второго клиента, а не воссоздание руками|
| Агент с tool calling | Config-слой на клиента (LLM-ключи, system prompt, какие тулы включены) без правки кода под каждого |
| Минимальный RAG | Чанкинг + ембединги уже робают но требует калибровки и тестирования |

## Порядок работ

Первым делом — безопасность автогенерации тулов, а не новые фичи. Тулы уже создаются из SQL-схемы и скоро коснутся боевой базы реального клиента, а не тестового стенда. Здесь нужен минимум: тулы с операциями записи по умолчанию выключены и требуют ручного подтверждения перед активацией, под агента заведён отдельный DB-пользователь с правами только на чтение там, где запись не нужна explicitно, и есть проверка, что мультитенантная изоляция держится под конкурентной нагрузкой, а не только в тестах на одном тенанте. Это не бюрократия ради галочки — если сгенерированный тул случайно даст доступ не к тем данным на реальном клиенте, это репутационный и потенциально юридический риск в самом начале трека.

Дальше — embed-виджет. Без него по сути нечего физически «интегрировать под ключ»: агент с RAG и тулами может прекрасно работать, но клиенту и его посетителям нужен видимый интерфейс. Здесь не нужна вся палитра из оригинального плана (темизация под все случаи, история в localStorage и т.д.) — нужен работающий Shadow DOM компонент со стримингом ответов и адекватным внешним видом на чужом сайте.

После виджета — минимальный config-слой на клиента. Не обязательно полноценная admin-панель с логами и аналитикой — достаточно того, что позволяет тебе самому (не клиенту) быстро завести нового клиента: указать его БД, LLM-провайдера, загрузить RAG-документы, включить нужные тулы. Полноценный Admin Dashboard из оригинального плана можно отложить до второго-третьего клиента, когда станет понятно, какие настройки реально приходится дёргать чаще всего.

Параллельно — упаковка деплоя: один docker-compose (core-сервисы всегда включены, профили только для мониторинга), `.env.example`, короткий внутренний runbook (не публичный DEPLOY.md для чужих новичков, а шпаргалка для себя), чтобы второй деплой занял часы, а не повторение всего пути с нуля.

Мессенджер-канал (Telegram или VK) в pre-final можно не включать вообще, если первый клиент явно не просит именно его. Это тот случай, где легко провалиться в разработку впрок — лучше добавить канал под конкретный запрос конкретного клиента, чем гадать заранее.

## Критерий готовности

Предфинал готов, когда весь стек можно направить на реальную БД и документы свежего клиента, настроить без правки исходного кода, поставить виджет на его сайт — и он отвечает на настоящие вопросы безопасно (тулы с записью выключены по умолчанию до ручного ревью), и держится минимум пару недель под реальными посетителями без того, чтобы ты каждый день что-то чинил руками.

## После первой продажи

Дальше — по уже обсуждённому приоритету: коннектор к Битрикс24, затем 1С (один-два типовых сценария обмена, не универсальный коннектор), затем VK и Telegram через общий слой «канал → tenant → агент». Именно в этот момент имеет смысл возвращаться к полноценному Admin Dashboard — когда будет понятно, под какие реальные сценарии его строить.
