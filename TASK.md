## ТЗ
- Работа в формате чата (на сайте аккуратный чат сбоку или подключение к боту в мессенджере, полноценное API для взаимодействия с агентом)
- Просмотр расписание студента
- Просмотр новостей вуза и расписания 
- Умение получить дисциплину их количество и конкретное отношение к студенту
- Генерация курсов на основе дисуиплины*
## Стек
- MCP-сервер — mcp Python SDK
- База — SQLite (sqlite3 из stdlib), фикстуры в JSON, возможно векторая бд в будущем
- Валидация — Pydantic (схемы ответов тулов)
- Поиск по материалам — TF-IDF ( sklearn), RAG в будущем
- Хост — локальная модель, что способна юзать тулы или облачное api (Groq, Mistral)

### Абстракции (примерный набросок)
```server.py           - регистрация тулов, точка входа
tools/
  student.py        - get_student, get_schedule
  disciplines.py    - get_disciplines, get_materials, search_materials
  course.py         - generate_course_plan
db/
  database.py       - SQLite-соединение, хелперы запросов
  models.py         - Pydantic-схемы (Student, Schedule, Material, CoursePlan)
fixtures/
  generate.py       - скрипт генерации тестовой базы
  *.json            - сырые данные

```
generate_course_plan не вызывает LLM — он собирает структуру из БД (темы → недели → материалы), модель оформляет это в текст сама. (в идеале полноценный агент а не просто mcp сервер)
search_materials — TF-IDF по полю content таблицы materials, возвращает топ-3 фрагмента или пожеланию модели если она не глупая.
### Переход в агента
Вся логика тулов остаётся без изменений, меняется только обёртка:
 PydanticAI : декоратор @agent.tool вместо MCP-регистрации, тот же api как бэкенд, + можно добавить свой чат поверх
Кастомное управление памятью более точнее и ответственное создание курсов и иных итерационых задач
В обоих случаях db/ и tools/ не трогаем
